Write a paper on a fight: Superman (New52) vs King Hyperion (Earth-4023)

Use this template to create one complete paper for any matchup: one-sided, close, rules-sensitive, or ability-sensitive.

## 1. Core contract
- Write the output in English.
- Use top-level sections `## 0..13` exactly once and in order.
- Keep `## 9`, `## 11`, and `## 12` even if the answer is `N/A`.
- Use `1-100` for all stats in section `3`.
- Use `X/10` only for verdict odds in sections `10` and `11`.
- Do not add extra top-level sections.
- Do not include meta commentary about what you changed or how you reasoned.

## 2. Research rules
### Source priority
1. `reddit.com/r/respectthreads`
2. `comicvine.gamespot.com/forums/gen-discussion-1/`
3. `marvel.fandom.com`
4. `dc.fandom.com`

### Source roles
- Respect threads and ComicVine feat/respect threads are the main proof sources.
- Marvel/DC Fandom are support sources only: use them for issue lookup, version checking, continuity checking, and page hunting.
- If a Fandom claim has no usable scan or image proof from a proof-oriented source, do not put it in the main list.

### Selection rules
- Do not stop at the first valid feat or win.
- Build a candidate pool first, then choose the final list.
- Main lists are curated, not exhaustive.
- If a strong claim has issue support but no usable scan, move it to a short backup appendix instead of the main list.

## 3. Required input
For each fighter provide:
- exact name and version
- short combat profile
- 9 matchup-relative stats in `1-100`
- top showcase feats
- top defeated / neutralized opponents
- source verdicts

### Main list format
Every main-list feat and every main-list win/result must include:
- name
- direct scan or image link
- exact source: title, issue, page if available
- short explanation

### Backup appendix format
Backup entries may omit the scan, but must include:
- claim name
- exact issue source
- short explanation
- note that no usable scan was found in primary proof sources

## 4. Estimation and selection
### Stats
- Estimate per source if needed.
- Map evidence to the 9 categories.
- Average into one final table.
- Round category values to whole numbers.
- Provide exact and rounded overall averages.

### Research selection workflow
1. Read at least 2 proof-oriented sources per character if available.
2. Build a candidate pool before selecting examples.
3. Remove wrong-version claims, low-proof summaries, duplicates, and weaker variants.
4. Rank candidates by:
   - version purity
   - proof quality
   - magnitude or prestige
   - directness or cleanliness
   - matchup relevance
   - coverage value
5. Select a curated top list.
6. Use backup appendix only for strong issue-backed claims without scan.

### Coverage rule
- Do not choose only the biggest-number feats.
- Where possible, the main feat list should cover at least 3 of these buckets:
  - scale
  - speed
  - durability
  - hax
  - battlefield influence

## 5. Scenario logic
- Lock exact versions and loadouts first.
- Test only material factors that can change the result.
- Use the minimum scenario count that fits the matchup.
- Stable matchup: one base verdict, section `11 = N/A`.
- One material factor: `2` scenarios.
- Two independent material factors: `4` scenarios.
- Every scenario must state assumptions, winner, odds, and short reasoning.
- Do not force conditional verdicts when the matchup is stable.

## 6. Required `final.md` structure
- `## 0 Hook` - central fight tension in 2-4 sentences.
- `## 1 Introduction and methodology` - source origin, synthesis method, and 9 categories.
- `## 2 Character introduction` - both fighters, profile, style, mentality, edges.
- `## 3 Averaged stats table` - 9-category table, exact and rounded averages, on-paper reading.
- `## 4 Origin and profile` - psychology and combat philosophy.
- `## 5 Powers and weaknesses` - tools, vulnerabilities, and swing factor.
- `## 6 Feats - quick review` - curated top showcase feats; default `5`, optional `6-8`, optional backup `max 3`.
- `## 7 Top defeated/neutralized opponents` - curated top showcase wins/results; default `5`, optional `6-8`, optional backup `max 3`; non-clean wins must be labeled.
- `## 8 Hypothetical fight - step by step` - opening, mid-fight, turning points, and `8.4` source verdict distribution.
- `## 9 Minority perspective` - strongest counter-position or `N/A`.
- `## 10 Base verdict` - baseline winner, `X/10`, short rationale.
- `## 11 Conditional verdict` - scenarios only if materially needed, otherwise `N/A`.
- `## 12 Why stats and verdict can diverge` - explanation or `N/A`.
- `## 13 Closing` - final confidence and biggest uncertainty.

## 7. Quality checklist
- Starts with `## 0. Hook`.
- Uses sections `## 0..13` exactly once.
- Has no extra top-level sections.
- Section `3` contains exactly 9 stat categories in `1-100`.
- Section `10` contains the base verdict.
- Section `11` is real scenarios or exact `N/A`.
- Section `8.4` tags verdicts as `explicit` or `derived`.
- Main feat list uses direct scan or image proof.
- Main win/result list uses direct scan or image proof.
- Backup appendix is used only for strong issue-backed claims with no usable scan.
- Lists are curated and not redundant.
- Versions and loadouts stay consistent.

## 8. Default assumptions
- Language: English.
- Stat scale: `1-100`.
- Verdict scale: `X/10`.
- Style: analytical writeup.
- If evidence is weak, use ranges and confidence language instead of fake precision.
- Keep scenario count minimal unless the prompt clearly requires more.
